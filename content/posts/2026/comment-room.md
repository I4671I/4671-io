静态网站没有数据库，也没有持续运行的后端程序。页面生成并发布之后，只是一组 HTML、CSS 和图片文件。因此，当我想给博客增加评论功能时，第一个问题并不是“评论框应该是什么颜色”，而是“评论内容要保存在哪里”。

## 静态页面的边界

浏览器可以显示表单，也可以让访客输入文字，但仅靠静态文件无法把输入永久保存下来。评论功能至少需要完成三件事：

1. 识别正在被评论的文章；
2. 保存评论者提交的内容；
3. 再次打开页面时读取并展示已有评论。

这些工作通常由外部评论服务或自己维护的服务器完成。对一个更新频率不高的个人网站来说，自己搭建后端意味着额外的数据库、登录验证、垃圾信息过滤和长期维护成本。

## 可以选择的方案

我目前考虑了以下几种方式。

### 使用 GitHub Discussions

以 GitHub Discussions 作为数据存储的评论系统很适合技术博客。评论公开透明，站长不需要单独维护数据库，还能使用 GitHub 已有的账号体系。

它的限制也很明显：访客通常需要 GitHub 账号。对于不熟悉 GitHub 的读者，这会增加参与门槛。

### 使用托管评论服务

另一种方式是接入专门的评论平台。这类服务通常提供审核、邮件提醒和垃圾评论过滤，接入时只需在文章模板中加入一段代码。

选择服务前，我会重点确认：

- 是否允许导出自己的评论数据；
- 免费额度是否适合个人网站；
- 是否会追踪访客或加载广告；
- 服务停止后，已有评论能否迁移。

### 暂时不设置评论区

这也是一个有效的选择。可以在网站上保留邮箱或其他联系方式，让真正想交流的读者直接来信。虽然交流不会公开展示，但网站会更轻量，也不必加载第三方脚本。

> 不添加某项功能并不代表网站不完整。有时，克制本身就是设计的一部分。

## 在模板中预留位置

如果以后决定接入评论系统，可以在文章布局底部加入一个独立区域：

```html
<section class="comments" aria-labelledby="comments-title">
  <h2 id="comments-title">评论</h2>
  <!-- 在这里加载评论组件 -->
</section>
```

把评论区域放在统一的文章模板中，而不是逐篇修改文章，可以确保所有文章保持一致。更换评论方案时，也只需要调整一个模板文件。

## 当前决定

现阶段我会先专注于写作，不急着开放评论。等网站有了稳定的访问和真实的交流需求，再选择一个允许导出数据、尽量尊重隐私的方案。

静态网站的优势正是简单。任何新功能，都应该在带来的价值明显大于维护成本时再加入。
